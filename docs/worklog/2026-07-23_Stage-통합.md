# 작업일지 — 2026-07-23 (담당: KimYunjae / Player 파트) · Stage 씬 통합

Ricochet-Apocalypse · Unity 6000.3.18f1 · 2D · Legacy Input Manager · URP · NavMeshPlus(2D NavMesh)

이번 작업: **NavMeshPathTest(적 몰려옴) + BulletTest(벽 튕김)을 참고해 Stage 씬에 합쳐 실제로 작동**하게 통합.

## 조사로 확인한 구조
- **적 몰려옴(NavMeshPathTest)**: 스포너 없이 미리 배치된 `Enemy_*`가 각자 `NavMeshAgent + EnemyAIModule + SpeedModule`로 tag `Player`를 추격. 장애물은 `NavMeshModifier(area=1, NotWalkable)`로 내비메시 카빙. `NavMeshSurface2D`(NavMeshPlus `NavMeshSurface + CollectSources2d`)가 베이크.
- **벽 튕김(BulletController)**: 총알 trigger가 Wall 레이어 콜라이더에 닿으면 `Bounce`(반사). `BulletPrefab`의 `wallLayerMask=Wall(8)`/`enemyLayerMask=Enemy(7)` **이미 설정돼 있음**(07-22 메모의 "미설정"은 갱신됨).
- **레이어**: Player=6, Enemy=7, Wall=8, Bullet=9.
- **두 종류의 적이 분리**돼 있었음: `EnemyPrefab`(체력 `Enemy:Entity` O, 이동 X) vs NavMeshPathTest 씬 적(이동 O, 체력 X) → **합본 필요**.
- **데미지 미배선**: `BulletDamageDispatcher.ApplyDamage`가 로그 스텁이라 `Enemy.TakeDamage`(현재 public) 미호출.
- **NavMeshPlus 2D 수집 규칙(핵심)**: `NavMeshBuilder2d`는 **`NavMeshModifier`가 붙은 오브젝트만** 소스로 수집. `useGeometry=PhysicsColliders`면 그 오브젝트의 `Collider2D`를 사용하고, 콜라이더 위치는 **attachedRigidbody 기준**. 또 `NavMeshSurface`는 **(-90,0,0) 회전** 필요.

## 후속 수정 (07-24) — 적이 안 움직이던 버그 해결

증상: 플레이 시 적이 NavMesh 경로(hasPath=PathComplete)는 잡는데 **velocity=0으로 이동 안 함**.

- **1차 오진**: NavMeshAgent 2D 자동 locomotion 버그로 보고 `Enemy2DAgentDriver`(agent.Move 수동 이동)를 임시 추가 → 오히려 엉뚱한 방향으로 밀어 navmesh 밖(Y 7.76 > navmesh 상단 7.33)으로 이탈. **폐기.**
- **근본 원인**: 정상 참고 씬 `NavMeshPathTest`의 적 Rigidbody2D는 **Dynamic(gravityScale 0)** 인데, 내가 합본 적을 만들 때 RB를 **Kinematic으로 바꿔** NavMeshAgent 자동 이동이 깨진 것. (초기 "안 움직임" 진단은 경로 계산 지연(~수십 초)을 못 기다린 탓도 있었음 — NavMeshPathTest도 8초엔 정지, 80초엔 추격 중이었다.)
- **해결**: `Enemy2DAgentDriver` 컴포넌트/스크립트 제거 + 적 RB를 **Dynamic(gravityScale 0, freezeRotation)** 으로 복원. 재생 검증: 적 4마리 전부 **vel≈2.0으로 플레이어 추격**(NavMeshPathTest와 동일 동작). 드라이버 불필요.
- 교훈: **NavMeshAgent 2D(NavMeshPlus) 이동체는 Rigidbody2D를 Kinematic이 아니라 Dynamic(gravity 0)** 으로 둘 것. 그리고 경로 형성엔 도메인 리로드 후 시간이 걸리니 **충분히 기다린 뒤** 판정할 것.

### 별개: 플레이가 아예 안 되던 원인(팀 공유)
- Play 진입 자체가 막혔던 건 게임 코드가 아니라 **MCP 하위 패키지(ivanmurzak inputsystem/tilemap/navigation)가 끌어온 깨진 core 0.86.0**의 컴파일 에러(`ProjectIdentity.DerivePortV2` 없음)였음. `Packages/manifest.json`+`packages-lock.json`을 직전 커밋(core 0.84.3)으로 되돌려 해결. **그 하위 패키지 3종은 업스트림 수정 전까지 다시 추가 금지.**

## 최종 해결 (07-24) — 적이 중앙 플레이어를 못 쫓던 문제 + 플레이어 2D화

증상: 적이 navmesh 경계(x=±5.33)로만 밀려 vel=0으로 굳고, 중앙 플레이어에 접근 못 함.

- 정밀 진단: 적 NavMeshAgent가 `dest=플레이어`인데 **rbPos≠transform(위치 desync), remainingDistance=0(도착 오판), velocity=0**. 즉 이 프로젝트 2D 설정에서 **NavMeshAgent 자동 locomotion이 경계 근처에서 신뢰 불가**(RB 제거·트리거화·warp·목적지 재설정 모두 실패). NavMeshPathTest가 되는 건 적이 scale 0.5라 경계에 안 닿아서였을 뿐, 근본은 agent locomotion.
- **해결**: `EnemyChaser2D` 를 **NavMeshAgent 비의존 직접 경로추종**으로 재작성 —
  `NavMesh.CalculatePath(현재→플레이어)`로 경로 계산 후, 다음 코너로 transform을 SpeedModule 속도만큼 이동, `NavMesh.SamplePosition`으로 navmesh 위로만(경계 이탈 불가). NavMeshAgent 제거, RB는 Kinematic(총알 트리거용).
  검증: 적 4마리 전부 중앙 플레이어(0,2)까지 **distP≈0.14로 도달**(스크린샷 확인).
- **navmesh 2D 확인(사용자 질문)**: NavMeshPlus로 -90 회전 베이크, 정점 전부 z=-0.08 단일 평면, 걷기영역이 XY 바닥 전체. **정상적인 2D navmesh 맞음** — 문제는 navmesh가 아니라 agent locomotion이었음.
- **Player 2D화(사용자 요청)**: Visual의 3D `MeshFilter/MeshRenderer`(구체) 제거 → 2D `SpriteRenderer`(built-in Knob, 빨강). (이건 적 문제의 원인은 아니었음 — Visual엔 콜라이더/물리 없음.)
- 교훈: **이 프로젝트에서 2D 적 추격은 NavMeshAgent 자동 locomotion 말고 `NavMesh.CalculatePath` + 직접 이동**을 쓸 것. navmesh 경로계산 자체는 정상.
- 남은 소소한 개선: NavMeshAgent 제거로 적끼리 회피(RVO)가 없어 플레이어 위에 겹쳐 쌓임 — 필요 시 간단한 분리(separation) 로직 추가 여지.

파일: 신규 `Assets/Scripts/EnemyChaser2D.cs`(직접 경로추종), 수정 `Assets/Scenes/Stage.unity`(적=EnemyChaser2D+Kinematic RB, NavMeshAgent/EnemyAIModule 제거, Player Visual 2D 스프라이트). 임시 `Enemy2DAgentDriver`는 폐기·삭제.

## 최종 마무리 (07-24) — 벽 통과/navmesh 밖 추격 수정 + 결정적 교훈(Run In Background)

사용자 보고: (1) 적이 장애물을 통과, (2) navmesh 밖까지 플레이어를 따라감.

- 원인분석 중 발견한 **가장 중요한 함정**: **Unity 에디터가 포커스를 잃으면 게임 루프가 멈춘다(Player Settings의 Run In Background 꺼짐).** 그래서 MCP로 조작하며 테스트하면 적이 "안 움직이는 것처럼" 보였고(실제론 게임이 일시정지), 이 때문에 원인을 여러 번 오진했다. `Application.runInBackground=true`로 켜자 즉시 정상 동작이 관측됨. → **MCP로 플레이 검증 시 반드시 Run In Background를 켤 것.**
- `EnemyChaser2D` 최종 로직(NavMeshAgent 없이):
  - `NavMesh.CalculatePath(내 위치→플레이어)`로 경로 계산, 다음 코너(corners[1])를 향해 이동(경로가 벽을 우회).
  - 매 스텝 `NavMesh.Raycast(현재→목표)`로 사이에 navmesh 경계(벽/바깥)가 있으면 **그 경계까지만** 이동 → **장애물 통과 불가 + navmesh 밖 이탈 불가**.
  - `PathInvalid`(플레이어가 navmesh 밖/도달불가)면 이동 안 함 → **navmesh 밖으로 안 따라감**.
  - 이동은 FixedUpdate에서. Kinematic RB면 `MovePosition`, 없으면 `transform.position`. (현재 Stage 적은 RB 없이 transform 이동 — 총알은 자신의 트리거+RB로 적 콜라이더 감지.)
- 검증(Run In Background 켠 상태):
  - 플레이어 중앙 → 적 4마리 코너에서 벽 우회해 **정확히 (0,2) 도달**(distP=0.00).
  - 플레이어를 navmesh 밖(12,12)으로 → 적은 **(0,2)에 머물고 navmesh 이탈 안 함**(onNavMesh=True).
- 남은 소소한 개선: 적끼리 분리(RVO) 없어 플레이어 위에 겹침. 필요 시 간단한 separation 추가.

## 최종 채택 (07-24) — 직접 스크립트 폐기, 정식 NavMeshAgent로 확정 (사용자 확인 완료)

- 사용자 지적/골드메탈 블로그(표준 2D NavMesh) 참고: 직접 짠 `EnemyChaser2D`(경로 흉내) 대신 **정식 `NavMeshAgent` 컴포넌트**로 재구성.
- Stage 적 최종 구성: `NavMeshAgent`(agentType 0, radius 0.4, speed 2, updateRotation/upAxis=false) + `EnemyAIModule`(코드에서 `agent.SetDestination(플레이어)`) + `SpeedModule` + `Enemy`(체력) + `EnemyStageBinder`(클리어 배선). **Rigidbody2D 없음**(에이전트가 transform 직접 구동 — 물리 충돌 간섭 제거). NavMeshSurface2D(NavMeshPlus, -90 회전, 베이크)는 유지.
- NavMeshAgent는 navmesh에 구속되므로 **벽 통과·navmesh 이탈이 구조적으로 불가능** → 사용자 요구(장애물 통과 금지, navmesh 영역만 인식) 충족.
- 검증(사용자 플레이 확인): 적이 벽 우회해 플레이어 추격, navmesh 밖으론 안 따라감.
- `Assets/Scripts/EnemyChaser2D.cs`(직접 스크립트)는 **삭제**.
- **재확인된 함정**: 앞서 "NavMeshAgent가 안 움직인다"던 진단은 전부 **Run In Background 꺼짐(에디터 비포커스 시 게임 정지)** 때문의 오진이었음. NavMeshAgent는 처음부터 정상이었다.
